01

Why it's actually slow

There are four distinct problems stacked on top of each other. Fixing any one of them helps. Fixing all four is how you get to <15 seconds.

Problem 1 — The entire remux happens before anything uploads
Biggest culprit
The current flow remuxes every single clip through FFmpeg WASM before the first byte hits R2. For 30 clips that's 30–60 seconds of phone CPU spinning before the upload progress bar even starts moving. FFmpeg WASM is also single-threaded on mobile — it can't remux two clips at once.

The fix is to interleave: remux clip 1 → immediately start uploading clip 1 → while that's uploading, remux clip 2 → etc. You get the upload clock ticking ~30 seconds earlier on large batches.
Problem 2 — Uploads are serial, one clip at a time
Major
The loop is for clip of clips: await uploadToR2(clip). Clip 2 cannot start until clip 1 is 100% done. On a typical home WiFi connection (10–30 Mbps upload), a single 200MB clip takes 55–165 seconds. Meanwhile your connection's bandwidth is idle waiting for the next loop iteration.

Uploading 3–4 clips concurrently would directly multiply effective throughput without changing your infrastructure at all.
Problem 3 — The user is a hostage to the loading screen
UX
Nothing visible happens until 100% of all clips have uploaded. The loading overlay says "Keep this screen open until it's done" — which for 30 clips can be 10–15+ minutes. A loading screen that long will get backgrounded. iOS will suspend the PWA's JavaScript. The upload fails. She has to start over.

The whole approach needs to flip: create the DB record immediately, navigate to the next screen within seconds (as soon as clip 1 is uploaded), and continue uploading the rest in the background while she's naming the scrapbook, browsing the workspace, or reviewing clips.
Problem 4 — No resume. Failure = restart from zero.
Reliability
If the upload is interrupted (screen lock, app backgrounded, connection dropped), there's no way to resume. The scrapbook record might not even exist in the DB yet. She has to pick all 30 clips again and start from scratch.

R2 supports multipart upload — splitting large files into chunks so a failure only loses the last chunk, not the whole file. Combined with an upload queue persisted to localStorage, you can survive a 30-second screen lock and pick right back up.
⏱
Current math for a 30-clip, 1GB batch on typical home WiFi 30 clips × ~1.5s remux = 45s remux wall · then 1GB ÷ ~15 Mbps upload = ~9 minutes serial upload. Total time on loading screen before seeing anything: 10+ minutes. That's not a UX problem. That's a broken product.

The Perception Fix — <15s to next screen

This is the highest-leverage change and it's almost entirely free to implement. The core idea: navigate the user to the next screen the moment clip 1 is available, and keep uploading everything else quietly in the background. The loading screen disappears in under 15 seconds regardless of how many clips she picked.

🎯
The key insight: clip 1 is all you need to get out of intake As soon as the scrapbook DB record exists and clip 1 has a video URL, the workspace can render. The other 29 clips uploading in the background are invisible to her until she actually navigates to them — by which point they'll be ready.

The new flow, step by step

  • NEW
    Remux clip 1 only — ~1–2 seconds. While this happens, skip the loading overlay entirely and show an inline progress indicator on the "Create Scrapbook" button (pulsing amber bar). User still sees the name sheet.
  • NEW
    Create scrapbook DB record + upload clip 1 in parallel — DB write is ~0.5s. Upload clip 1 at full bandwidth. This is the one clip you wait for. Clips 2–30 haven't uploaded yet.
  • NEW
    Navigate immediately — as soon as clip 1's video_url is written to the DB, navigate to /scrapbook/:id (detail screen) or /scrapbook/:id/edit (workspace). Total elapsed time: remux + clip 1 upload. For a 50MB first clip: ~5–12 seconds.
  • NEW
    Global upload context drains the queue — a React context (see architecture below) continues remuxing + uploading clips 2–30 with 3 concurrent uploads. A persistent amber upload banner on the detail/workspace screen shows "Uploading 28 more clips…" — non-blocking, tappable to expand progress.
  • existing
    Wake lock stays active — already implemented. Keeps screen on while upload context is draining. Re-acquired on visibilitychange if she tabs back in.
  • NEW
    Clip cards show "uploading" state — clips that haven't uploaded yet show a pulsing amber shimmer in the workspace clip strip. Tapping a not-yet-uploaded clip shows "Still uploading… tap again when ready" toast instead of navigating to playback.

Upload context architecture

A new UploadContext (alongside existing AuthContext) owns the upload queue. It survives navigation — IntakeScreen hands off the queue and immediately returns control to the router.

The Speed Fix — actually faster

Perception and reality are separate problems. Even after the perception fix, you want the background queue to drain as fast as possible so the workspace is fully loaded quickly. Here are the levers ranked by impact.

1
Concurrent uploads (3–4 in parallel)
Change the serial for...await loop to a concurrency-limited pool. If each clip takes 20 seconds and you run 4 at once, your effective throughput is 4× higher on a fast connection. No infrastructure changes required — the Cloudflare Worker and R2 handle concurrent PUT requests natively.

Start with 3 concurrent. Too many simultaneous large uploads on cellular can saturate the connection and actually slow things down — 3–4 is the sweet spot.
Low effort 4× speedup
2
Interleave remux + upload (pipeline clip processing)
Instead of remux-all → upload-all, process clips in a pipeline: remux clip N → immediately start uploading clip N → begin remuxing clip N+1. Since FFmpeg WASM is single-threaded, only one remux runs at a time — but uploads run concurrently alongside it. This eliminates the dead ~45-second remux wall before any upload begins.
Medium effort −45s on large batches
3
Skip remux for clips that don't need it
FastStart remux is valuable for playback streaming but it's optional — videos play fine without it, just with a slight initial delay. Consider skipping remux entirely for clips >200MB or when the queue is large, and relying on R2's CDN edge caching to mask the moov-atom issue. Photos already skip remux. Extend this logic: if file.size > 150MB, upload raw.

Alternatively: upload raw first, then remux server-side via a Cloudflare Worker that calls an external FFmpeg service. Upload time drops to zero wait. (Requires new infrastructure.)
Low effort −1–2s per clip
4
Presigned direct-to-R2 uploads
Currently: client → Cloudflare Worker → R2 (two hops). With presigned URLs: client → R2 directly (one hop). The Worker already has a generatePresignedPut() function but it's not wired to a route or used by the frontend.

The latency difference is modest on large files (the Worker is already very fast) but it removes the Worker as a potential bottleneck and removes one TLS handshake. Worth doing but not the biggest win.
Medium effort Modest win
5
Client-side video compression before upload
Already in the parking lot in DECISIONS.md. Compressing 1GB to, say, 300MB would reduce upload time by 70% — the single biggest lever if you can pull it off. The challenge: re-encoding on iPhone via FFmpeg WASM takes 5–20× real-time (a 30s clip could take 5–10 minutes to compress). Not viable client-side for large batches.

The right path: upload raw → trigger server-side transcode (e.g. Cloudflare Stream, AWS MediaConvert, or a small VPS running FFmpeg). The compressed version replaces the raw in R2 asynchronously. Significant infrastructure investment but real impact.

The Reliability Fix — no more starting over

Even with the perception fix, a background upload for 1GB of video takes several minutes. If the phone goes to sleep or loses WiFi, those uploads should be resumable — not restarted.

iOS background problem — honest assessment
Hard constraint
Wake lock prevents the screen from turning off while the app is in the foreground. But if she switches to Messages or the screen locks anyway (low battery mode ignores wake lock), iOS Safari suspends JavaScript execution. Background Fetch API and Service Worker Background Sync have extremely limited iOS PWA support.

You cannot fully solve background uploads in a PWA on iOS. What you can do is make the failure recoverable instead of catastrophic — so she loses 30 seconds of progress, not 15 minutes.
Multipart upload for large clips (>50MB)
Recommended
R2 supports S3-compatible multipart upload. Your Worker already has all the crypto infrastructure to generate AWS4-signed requests. The flow:

1. CreateMultipartUpload → get an uploadId back from R2
2. UploadPart in parallel — split 200MB file into 10MB chunks, upload 5 chunks at once
3. CompleteMultipartUpload → R2 assembles the file
4. If interrupted after chunk 8 of 20: resume from chunk 9, not from chunk 1

Store { uploadId, completedParts, key } in localStorage. On app reopen, check for in-progress uploads and offer to resume.
Upload queue persistence in localStorage
Recommended
The UploadContext should serialize its pending queue to localStorage on every state change. On app startup, if there's a pending queue for a scrapbook that exists in the DB, offer to resume: "You have 18 clips still uploading from your last session. Continue?"

Note: the raw file Blobs themselves are not serializable — you can't save them to localStorage. What you CAN save: which clip IDs were already successfully uploaded (their video_url is in the DB). On resume, you only need to re-pick the files for the unfinished clips — which means she'd need to re-select from camera roll, but only the remaining ones, not all 30.

The better experience: a "Resume Upload" banner that explains she needs to re-select the remaining clips. Annoying but far better than starting over entirely.
💡
The real solution to "app goes to sleep mid-upload" The perception fix is the actual solution here. If the loading screen is gone in <15 seconds, she's actively using the app — browsing the workspace, naming things, editing clip 1 — and iOS won't background the app. The background upload drains while she's engaged. The sleep problem mostly goes away when you stop requiring her to watch a progress bar for 10 minutes.
05

Is it just more money?

Short answer: no. The bottlenecks in your stack are architectural, not subscription tier. Here's the honest breakdown.

Cloudflare R2 plan upgrade?
No
R2's upload speed is not tier-gated. You're already on the right product. R2 ingress (upload) is free. The bottleneck is client internet speed and serial architecture, not R2 throughput.
Cloudflare Workers paid plan?
No
Your Worker handles the upload in microseconds (it just passes the stream through). The Worker is not the bottleneck. Paid plan would only help if you hit CPU time limits, which you won't for streaming uploads.
Supabase Pro (already have it)?
Keep it
Supabase Pro is correct for your DB reads/writes. The DB round-trips (create scrapbook, insert clip rows) are fast — <200ms each. Not a bottleneck.
Cloudflare Stream?
Maybe v2
$5/1000 min stored + $1/1000 min delivered. Would give you adaptive bitrate streaming, server-side compression, and a proper video CDN. Real quality upgrade, real cost. Probably overkill for a family app right now.
✅
All three highest-impact fixes cost $0 in new infrastructure Concurrent uploads, pipelined remux, and the optimistic navigation / upload context are pure code changes. The existing Worker, R2, and Supabase stack handles them without any config changes. Estimated implementation time: 1–2 focused sessions.

Build roadmap

Ordered by impact-to-effort. Do Phase 1 first — it gets you to the 15-second goal. Phase 2 makes the background drain faster. Phase 3 is polish and reliability.

Phase 1 — The 15-second goal · 1 session

  • IntakeScreen
    Remux + upload clip 1, then navigate — change handleCreate(): remux clip 1 only → create DB record → upload clip 1 → insert clip 1 row → call startBackgroundUpload(clips.slice(1)) → navigate. Remove the full-screen upload overlay for the first clip; show an inline progress bar on the Create button.
  • New file
    Create src/context/UploadContext.jsx — holds the async upload queue. Exports useUpload() hook. Renders a persistent non-blocking amber banner ("Uploading 27 more clips…") in App.jsx above the router outlet when isActive. No loading screen. No blocking navigation.
  • WorkspaceScreen
    Unready clip states — clips without a video_url yet show a pulsing shimmer overlay in the horizontal clip strip. Tapping shows a gentle amber toast: "This clip is still uploading. Try again in a moment."
  • ScrapbookDetailScreen
    Upload progress awareness — if useUpload().scrapbookId === id, show a small "Uploading X clips" subtitle under the scrapbook name. Disappears when done.

Phase 2 — Faster background drain · 1 session

  • UploadContext
    Concurrent upload pool (3 slots) — replace the serial queue with a pool that keeps 3 uploads in-flight simultaneously. Use a simple counter + promise chain pattern. No new libraries needed.
  • UploadContext
    Pipelined remux — the queue processor remuxes the next clip while the current clip is uploading. Since FFmpeg WASM is single-threaded, only one remux runs at a time — but it runs concurrently with active uploads instead of blocking them.
  • IntakeScreen
    Skip remux for clips >150MB (optional) — large clips upload raw. FastStart optimization only applies to clips ≤150MB where the remux cost is low relative to the file size. Adjust threshold based on Joelle's feedback.

Phase 3 — Resilience · future sprint

  • Worker + r2.js
    Multipart upload for clips >50MB — add /multipart/create, /multipart/upload-part, /multipart/complete routes to the Worker. Update r2.js to use multipart automatically for large files. Chunked transfer means a 30-second interruption only loses one 10MB chunk, not the whole clip.
  • UploadContext
    Persist upload progress to localStorage — serialize the list of completed clip IDs on every success. On app restart with a stale queue, show a "Resume upload" banner. She won't need to re-select all 30 clips — only the ones that didn't make it.
  • Worker
    Expose presigned URL endpoint — wire up the existing generatePresignedPut() to a POST /presign route. Update r2.js to use presigned direct-to-R2 PUT for large files. Removes one network hop for the upload body.
🎬
After Phase 1 alone: Joelle picks 30 clips, hits Create Scrapbook, and sees the workspace in 8–15 seconds. The other 29 clips upload quietly while she's looking at clip 1, editing, naming things. No loading screen. No "keep this open." No starting over. That's the goal. Phase 2 makes it even faster. Phase 3 makes it bulletproof.
